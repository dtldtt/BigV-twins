## 我的评价与建议

### 推荐：方案 C（原文 + brief 混合）

**理由**：

1. **信息零损耗** — LLM 能看到完整原文，不存在 brief 压缩导致的信息衰减
2. **结构最清晰** — brief_json 作为「已标注的路标」帮 LLM 快速定位重点，减少在长文中「迷路」
3. **速度最快**（102s < A的114s < B的154s）— 看似反直觉，但 brief 标注让 LLM 更高效地处理原文
4. **分歧识别最好** — 找到 2 条有价值的分歧，且分析更深入
5. **共识更全** — 4 条 vs A 的 3 条

**方案 B 也很好**，但没有 C 的结构优势。如果 token 成本敏感，B 是次优选择。

**方案 A 不推荐**，信息衰减太严重，特别是分歧识别能力明显不足。

### Token 成本估算

- 方案 C 每天：~23K input + ~5K output ≈ 28K tokens
- 5 个博主 × ~5000 字原文 = ~35K tokens 峰值
- 走 Qoder performance（月付），无额外成本

## 测试过程中发现的 Bug

测试过程中发现并修复了一个关键 bug：

**`_open_zhihu_ro()` 的 `immutable=1` 导致读不到新数据**

- `immutable=1` 让 SQLite 缓存旧的 DB 文件快照，zhihu crawler 新爬的帖子（6 月 5-8 日）完全不可见
- 首次测试三个方案输入全为 0 字符，LLM 返回"没有数据"
- 修复：去掉 `immutable=1`，改为 `mode=ro`（只读但不缓存）
